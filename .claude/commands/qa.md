# アプリ品質保証チェック（QA）

あなたはオーケストレーターです。以下のワークフローを**すべて subagent に委託**して並列実行してください。

## 対象

$ARGUMENTS

引数が空の場合は `scripts/` 配下の `*_app.py` を一覧表示し、ユーザーに選択を求める。

---

## ワークフロー

### Step 0: 対象ファイルと関連ファイルの特定

対象アプリファイルに対して:
- 同じディレクトリ内の関連ファイルを自動検出する（`*help_content.py`, `*config.py`, `generate_*_scenario.py` 等）
- `docs/` 配下で対象アプリに関連するマニュアル・ドキュメントを検出する
- データファイル（`data/` 配下の CSV 等）を検出する

### Step 1〜3: 3層チェック（並列 subagent で実施）

#### Layer 1: 数値・定数の一貫性チェック
- アプリファイル内の全定数（金額・閾値・日数・ラベル等）を抽出する
- 関連ファイル（ヘルプ・マニュアル・シナリオ）内の数値記載を全抽出する
- 両者を突合し、不一致を検出する
- 不一致があれば修正する（アプリ側が正、ドキュメント側を修正）
- **出力:** 突合テーブル（項目 / アプリの値 / ドキュメントの値 / 一致・不一致）

#### Layer 2: コードのスコープ安全性・全UI状態チェック
- `py_compile` + `ast.parse` で構文チェックを実行する
- 以下の4パターンを検出する:
  - **(a)** 変数が条件ブロック内でのみ定義され、ブロック外で参照されている（NameError リスク）
  - **(b)** 特定の表示モードでしか成立しないロジックが、他のモードでも実行される
  - **(c)** else ブロックからのインデント漏れ（if/else 直後のコードが else の外に出ている）
  - **(d)** 複数モード間で dict のキーセットに差異がある（KeyError リスク）
- UI の全モード切替（タブ、ラジオボタン、セレクトボックス等）を列挙し、全組み合わせでの変数参照を検証する
- **出力:** 問題テーブル（タブ名 / 行番号 / パターン(a-d) / 変数名 / 修正内容）

#### Layer 3: 連動ドキュメント整合性チェック
- アプリ内の機能名・タブ名・ラベル・選択肢と、ドキュメントの記述を照合する
- コードにあってドキュメントにない機能、またはその逆を検出する
- 古い用語・旧機能名が残っていないか確認する
- **出力:** 不一致テーブル（ファイル / 箇所 / 現在の記述 / 正しい記述）

### Step 4: 統合レポートと修正適用

3層の結果を統合し、以下の形式で報告する:

```
## QA完了報告
| レイヤー | チェック項目数 | 結果 | 要修正件数 |
|----------|-------------|------|-----------|
| L1 数値一貫性 | N | ✅/❌ | n件 |
| L2 スコープ安全性 | N | ✅/❌ | n件 |
| L3 ドキュメント整合性 | N | ✅/❌ | n件 |
```

- 修正があった場合: 構文チェック（py_compile + AST parse）を再実行して通過を確認する
- 新しいバグパターンを検出した場合: `memory/project_qa_lessons.md` に追記する
- 最後に git commit + push を実行する
