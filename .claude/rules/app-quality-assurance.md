# アプリ品質保証ルール

## 適用条件
- Pythonアプリケーションファイル（*.py）を変更するとき、このルールに従う

## 1. 単一ソース原則（DRY）
- 定数（金額・閾値・ラベル等）は1箇所で定義し、他はすべて参照する
- 定数を変更した場合、Grepでその値を参照している全ファイルを検索し、不一致を修正する
- 同じ値が2箇所以上ハードコードされていたら、共通定数への集約を提案する

## 2. 連動更新（双方向）
- **コード → ドキュメント**: コードを変更したら、以下の関連ファイルも必ず確認・更新する:
  - ヘルプ・使い方コンテンツ（*help_content.py 等）
  - 運用マニュアル・ドキュメント（docs/ 配下）
  - デモシナリオ・台本
- **ドキュメント → コード**: DOCX ファイルが変更された場合（git diff で検出）、`/sync-docs` コマンドを実行してアプリとの整合性を確認する。新しい用語・数値がアプリに反映されていなければ修正を提案する
- 連動更新が必要な場合、指示がなくても自動的に修正する
- 修正した連動ファイルの一覧をユーザーに報告する

### git push 時の DOCX 整合性チェック
- git push に .docx ファイルが含まれる場合、push 後に `/sync-docs` の簡易版チェックを自動実行する
- DOCX の新しい用語がアプリに未反映の場合、ユーザーに「アプリにも反映しますか？」と確認する
- DOCX が「正」、アプリが「従」の関係（ユーザーが直接修正した内容を尊重）

## 3. スコープ安全性
- if/elif/else ブロック内で初めて定義する変数は、すべてのブランチで定義されているか確認する
- 条件ブロックの外で使う変数は、ブロックの前にデフォルト値を定義する
- else ブロック直後のインデントが正しいか（漏れ出していないか）必ず確認する
- UIの表示モードごとに異なるデータ構造を使う場合、共通のキーセットを保証する

## 4. 全状態テスト
- UIにモード切替・タブ・選択肢がある場合、すべての組み合わせで動作を確認する
- 変更後は必ず実行: `py_compile` による構文チェック + `ast.parse` による構造チェック
- Streamlitアプリの場合、`st.session_state` の初期化漏れがないか確認する

## 4.5 統合テスト（状態遷移の安全性）
**背景**: 純粋関数のユニットテスト＋構文チェックだけでは、「セクション切替時にデータ未定義でクラッシュ」のような**状態遷移バグ**を検出できない。これを防ぐため、以下を必ず実施する。

### 自動検証（test_app_integration.py）
- `bed_control_simulator_app.py` を変更したら、`tests/test_app_integration.py` を必ず実行する
- このテストはAST解析で以下を検証する:
  - **タブ名整合性**: `_tab_idx["..."]` で参照される全タブ名が `_ALL_TAB_NAMES` に存在するか
  - **変数スコープ安全性**: `df`, `summary`, `days_in_month` 等がif/else全ブランチで定義されるか
  - **st.stop()安全性**: `st.stop()` がデータ不要セクション（制度管理等）をブロックしないか
  - **パスワードガード**: データ入力・エクスポートタブに認証ガードがあるか
  - **session_state安全性**: `st.session_state.X` への直接アクセスが適切にガードされているか

### 手動確認（アプリ変更時のチェックリスト）
- [ ] 全5セクション（ダッシュボード/意思決定/制度管理/データ管理/HOPE）への切替が動作するか
- [ ] シミュレーション未実行状態で各セクションに切替えてエラーが出ないか
- [ ] 実績データなし状態で各セクションに切替えてエラーが出ないか

### 新しいタブ・セクションを追加する場合
- `_ALL_TAB_NAMES` にタブ名を追加する
- セクション→タブマッピングに追加する
- `_needs_sim_data` にそのセクションがデータ依存か明示する
- `test_app_integration.py` のテストが通ることを確認する

## 5. リグレッション影響分析（最重要）
過去に苦労して実装した機能が、新しい修正で知らぬ間に壊れる現象を **リグレッション** と呼ぶ。これを防ぐため、**すべての修正** で以下を実施する。

### 修正前（Plan 段階）
- 修正対象の関数・変数が**どこから呼ばれているか**を Grep で確認する
- 呼び出し元ごとに「この修正で影響を受けるか / 受けないか」を明示的に列挙する
- 影響範囲が3箇所以上なら、修正方針をユーザーに報告してから進める

### 修正後（Check 段階）
- `python3 scripts/hooks/smoke_test.py` を必ず実行し、ベースライン期待値との一致を確認する
  - 主要計算（稼働率・単体目標・均等努力Δ・rolling LOS）が期待範囲内か
  - 失敗した場合、その変化が**意図した修正**か**意図せぬリグレッション**かを判定する
  - 意図した変更であれば `EXPECTED` を更新、そうでなければ修正を見直す
- `docs/admin/regression_test_checklist.md` の関連機能を手動で確認する

### コミット時
- pre-commit hook（`scripts/hooks/qa-precommit.sh`）が自動でスモークテストを実行する
- `bed_control_simulator_app.py` / `bed_data_manager.py` / デモCSV が変更された時のみトリガー
- `--no-verify` でフックをスキップして強引にコミットすることは **絶対に禁止**
