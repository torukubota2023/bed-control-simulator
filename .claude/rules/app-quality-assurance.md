# アプリ品質保証ルール

## 適用条件
- Pythonアプリケーションファイル（*.py）を変更するとき、このルールに従う

## 1. 単一ソース原則（DRY）
- 定数（金額・閾値・ラベル等）は1箇所で定義し、他はすべて参照する
- 定数を変更した場合、Grepでその値を参照している全ファイルを検索し、不一致を修正する
- 同じ値が2箇所以上ハードコードされていたら、共通定数への集約を提案する

## 2. 連動更新
- コードを変更したら、以下の関連ファイルも必ず確認・更新する:
  - ヘルプ・使い方コンテンツ（*help_content.py 等）
  - 運用マニュアル・ドキュメント（docs/ 配下）
  - デモシナリオ・台本
- 連動更新が必要な場合、指示がなくても自動的に修正する
- 修正した連動ファイルの一覧をユーザーに報告する

## 3. スコープ安全性
- if/elif/else ブロック内で初めて定義する変数は、すべてのブランチで定義されているか確認する
- 条件ブロックの外で使う変数は、ブロックの前にデフォルト値を定義する
- else ブロック直後のインデントが正しいか（漏れ出していないか）必ず確認する
- UIの表示モードごとに異なるデータ構造を使う場合、共通のキーセットを保証する

## 4. 全状態テスト
- UIにモード切替・タブ・選択肢がある場合、すべての組み合わせで動作を確認する
- 変更後は必ず実行: `py_compile` による構文チェック + `ast.parse` による構造チェック
- Streamlitアプリの場合、`st.session_state` の初期化漏れがないか確認する
