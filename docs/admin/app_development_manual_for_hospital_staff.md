# 院内職員のためのアプリ開発マニュアル

## ― 現場のニーズから生まれるアプリ、それを自分たちで作る ―

**作成者:** 副院長 久保田 徹（おもろまちメディカルセンター）
**対象:** 院内の一般職員（看護師・事務・医師・MSW・薬剤師など職種問わず）
**前提知識:** パソコンでメールとWebが使える程度でOK
**目的:** 現場のニーズに合わせて独自にアプリを開発・検証・運営し、結果を出せるようになる

---

# はじめに — なぜ自分たちでアプリを作るのか

## これまでの常識

「アプリを作りたい」と思ったら、従来はこのどれかでした：

1. IT部門に依頼する → 優先順位待ち、半年〜1年
2. 業者に発注する → 数百万円〜数千万円
3. 市販のソフトを購入する → 現場に合わない、高い
4. あきらめる → 多くの人がここ

## 今の時代は違います

AIアシスタント（Claude Code など）と、無料のクラウドサービスの登場で、**プログラミング経験がない人でも、現場のニーズに合ったアプリを自分で作れる時代**になりました。

本マニュアルは、その具体的な方法を、超初心者向けに解説するものです。

## 本マニュアルの大前提

- **失敗しても大丈夫** — 無料ツールだけで作れるので、金銭的リスクはゼロ
- **小さく始める** — 完璧を目指さず、動くものから始める
- **AIは相棒** — 「何がわからないかもわからない」状態でも、AIに聞けば教えてくれる
- **現場が主役** — 業者に任せるのではなく、現場が自分たちで作って運用する

## このマニュアルの範囲

本マニュアルで扱うのは「**院内の非機微情報を扱う業務支援アプリ**」です。具体的には：

### ✅ 扱ってよいもの

- 病床稼働率のシミュレーション
- 運用ルールの可視化ツール
- スタッフ教育用アプリ
- 会議資料の自動生成
- KPI計算ツール
- 勤務シフトの試算
- 業務フローのチェックリスト

### ❌ 扱ってはいけないもの（このマニュアル範囲外）

- **患者の個人情報を含むもの**（氏名・ID・病名など）
- **電子カルテと直接連携するもの**
- **保険請求システムに影響するもの**
- **院外ネットワークと通信して患者データを送るもの**

**原則: 患者個人情報は一切扱わない。集計値のみ。**

# 第1章 準備するもの

## 必要なもの（すべて無料）

| 項目 | 内容 | 費用 |
|---|---|---|
| パソコン | Mac または Windows | — |
| インターネット | 通常の接続 | — |
| GitHub アカウント | ソースコードを保管する場所 | 無料 |
| Streamlit Cloud アカウント | 作ったアプリを公開する場所 | 無料 |
| Claude（またはClaude Code） | AIアシスタント | 個人プランあり |
| Google アカウント | GitHub/Streamlit のログインに使う | 無料 |

## 1-1. GitHub アカウントを作る

GitHubは、作ったアプリのソースコード（プログラム本体）を保管するクラウドサービスです。世界中のプログラマーが使っている定番のサービスで、無料で使えます。

### 手順

1. https://github.com/signup にアクセス
2. メールアドレスを入力
3. パスワードを設定
4. ユーザー名を決める（後から使うので覚えやすいものに）
5. 認証メールが届くので、確認コードを入力

**所要時間: 約5分**

### よくある質問

- **Q: 英語だらけで不安です**
- A: ブラウザの翻訳機能（Chromeなら右クリック → 日本語に翻訳）を使えばOK

## 1-2. Streamlit Cloud アカウントを作る

Streamlit Cloud は、作ったアプリをインターネットで公開するためのサービスです。GitHub アカウントがあれば、そのままログインできます。

### 手順

1. https://streamlit.io/cloud にアクセス
2. 「Sign up」または「Continue with GitHub」をクリック
3. GitHub アカウントで連携（許可を求められたら許可）

**所要時間: 約3分**

## 1-3. Claude を使えるようにする

アプリを作る際のAIアシスタントです。2つの使い方があります：

### 方法A: Claude.ai を使う（最も簡単）
1. https://claude.ai にアクセス
2. Google アカウントでログイン
3. ブラウザ上で対話するだけで使える

### 方法B: Claude Code を使う（より強力）
Claude Code は、パソコン上で動くAIアシスタントで、ファイル操作やコマンド実行までAIに任せられます。少し設定が必要ですが、作業効率が格段に上がります。

本マニュアルでは **方法A（Claude.ai）** を前提に解説します。慣れてきたら方法Bに移行してください。

## 1-4. このマニュアルを読むために覚える用語

最初は少しだけ専門用語が出てきます。以下を覚えればOKです：

| 用語 | 意味 | 例え |
|---|---|---|
| **リポジトリ（repo）** | ソースコードを保管する箱 | プロジェクトフォルダ |
| **コミット** | 「このバージョンを保存」する操作 | Wordの上書き保存 |
| **プッシュ** | 保存した内容をGitHubにアップロード | クラウドにアップロード |
| **プル** | GitHub から最新版を取得 | クラウドからダウンロード |
| **デプロイ** | アプリを公開すること | Webサイトを公開するのと同じ |
| **Streamlit** | Pythonで簡単にアプリを作れる道具 | PowerPointでチラシを作るような感覚 |
| **Python（パイソン）** | プログラミング言語の一種 | 日本語の一種、のような関係 |

**重要:** これらの用語を完全に理解する必要はありません。「こういう名前のものがある」とわかれば十分です。

# 第2章 アイデアを見つける（最も重要な最初の一歩）

## 「アプリを作ろう」と思う前に

**最も失敗するパターン**は、最初に「こんなアプリを作ろう」と決めてから現場を見ることです。

**成功するパターン**は、**現場の「不便」に気づくこと**から始まります。

## 不便に気づくための質問

毎日の業務で、以下を自問してみてください：

- 「この計算、毎回Excelで手作業…面倒だな」
- 「同じ質問に何度も答えてる」
- 「この会議資料、作るのに毎回3時間かかる」
- 「数字がバラバラで、全体像が見えない」
- 「新人に教えるのが大変」
- 「ミスが起きやすいところがある」

これら全部が**アプリのタネ**です。

## 実例: ベッドコントロールシミュレーターの場合

当院では、こんな不便がありました：

### 不便1: 稼働率と収益の関係が見えない
> 「稼働率を上げよう」と朝会で言われても、1%上がるといくら違うのか誰も答えられなかった。

### 不便2: What-if がない
> 「C群の患者を3名退院させたら月末の平均在院日数はどうなる？」という質問に、Excelで30分かけて計算していた。

### 不便3: 病棟間の違いが見えない
> 5Fと6Fで課題が違うのに、全体の数字しか見ていなかった。

**これらが「ベッドコントロールシミュレーター」の開発動機**になりました。

## アイデアを書き出す（5分）

以下のフォーマットで書いてみましょう：

```
【現場の不便】
（例: 月末に平均在院日数の見込みがわからず、毎回焦る）

【誰が困っているか】
（例: ベッドコントロール担当の看護師、医事課）

【どうなれば嬉しいか】
（例: 月の途中でも、月末予測が自動表示される）

【必要なデータ】
（例: 日次の在院患者数、新入院数、退院数）

【個人情報は含むか】
（例: 含まない。集計値のみ）
```

**ポイント:**
- 「個人情報を含まない」ことを必ず確認
- データが複雑すぎない（最初は3〜5項目まで）
- 結果が数字またはグラフで出せるもの

## アイデアを小さく分解する

「大きな完成形」を目指さず、**最初の1機能** だけに絞ります。

### 例: ベッドコントロールシミュレーター

**最終形（今）:** 12タブの巨大アプリ（日次推移、フェーズ構成、運営分析、意思決定ダッシュボード、What-if、医師別分析、HOPE連携など）

**最初のバージョン（v0.1）:**
- 日次の在院患者数を入力
- 稼働率を計算して表示
- それだけ

**たったこれだけ**です。最初は。

# 第3章 AIと対話して最初のアプリを作る

## AIとの基本的な対話方法

Claude.ai を開いて、以下のように話しかけます。**丁寧な日本語でOK**です。プログラミング用語は不要です。

### 最初に伝えるべき3つのこと

1. **何を作りたいか**（目的）
2. **何を入力して**（入力）
3. **何を出力してほしいか**（出力）

### 会話例（コピペして使ってOK）

```
Claudeへ:

私は病院の看護師で、プログラミング経験はありません。
現場の業務支援アプリを作りたいと思っています。

作りたいもの:
病床稼働率を計算する簡単なアプリです。

入力:
・在院患者数（例: 42名）
・総病床数（例: 47床）

出力:
・稼働率（%）
・目標の90%までの差（例: 「あと1名でクリア」）

条件:
・Streamlit で作りたい
・患者個人情報は扱わない
・ブラウザで動くシンプルなもの
・私はプログラミング初心者です

完全なコードを見せてください。
また、それを動かすための手順も1から教えてください。
```

## AIから返ってくるもの

Claudeは、上記のような依頼に対して：

1. **完全なPythonコード**（例: `app.py` というファイル名で保存する内容）
2. **保存する場所の説明**
3. **動かす手順**

を順番に教えてくれます。

## コードをパソコンに保存する

AI が出してくれたコードを、実際にパソコンに保存します。

### Mac の場合

1. 「テキストエディット」を開く（Spotlight検索で入力）
2. 「フォーマット」→「標準テキストにする」を選ぶ
3. AIから受け取ったコードをコピーして貼り付け
4. 「ファイル」→「保存」
5. ファイル名を `app.py` にする（拡張子 `.py` が重要）
6. 保存場所は「書類」→新しいフォルダ「my-app」に

### Windows の場合

1. 「メモ帳」を開く
2. コードを貼り付け
3. 「ファイル」→「名前を付けて保存」
4. ファイル名を `app.py` にする
5. 「ファイルの種類」を「すべてのファイル」に変更
6. 文字コードは「UTF-8」を選ぶ
7. 保存場所は「ドキュメント」→新しいフォルダ「my-app」に

## 動かす準備: Python のインストール

Pythonは、アプリを動かすために必要な「エンジン」です。一度入れれば、あとは気にしなくていい存在です。

### Mac の場合

1. https://www.python.org/downloads/ にアクセス
2. 「Download Python 3.x.x」をクリック
3. ダウンロードしたファイルを開いてインストール

### Windows の場合

1. 同じく https://www.python.org/downloads/
2. 「Download Python 3.x.x」
3. ダウンロードしたファイルを開く
4. **重要: 「Add Python to PATH」にチェックを入れる**
5. 「Install Now」

## Streamlit をインストールする

Pythonが入ったら、次にStreamlitという道具を入れます。

### Mac の場合

1. 「ターミナル」を開く（Spotlight検索）
2. 以下を入力してEnter:
```
pip install streamlit
```
3. 「Successfully installed...」と出れば完了

### Windows の場合

1. 「コマンドプロンプト」を開く（スタートメニューから）
2. 同じコマンド:
```
pip install streamlit
```

## アプリを動かす

準備完了です。いよいよアプリを起動します。

1. ターミナル（Mac）またはコマンドプロンプト（Windows）を開く
2. 保存したフォルダに移動:
```
cd ~/Documents/my-app    (Mac)
cd %USERPROFILE%\Documents\my-app    (Windows)
```
3. アプリを起動:
```
streamlit run app.py
```
4. ブラウザが自動で開いて、あなたのアプリが動きます！

## 動かなかったら

エラーメッセージが出たら、そのメッセージ全文をコピーして、Claudeに貼り付けます：

```
以下のエラーが出ました。直してください。

[エラーメッセージをここに貼り付け]
```

Claudeが原因と解決方法を教えてくれます。

**ポイント: エラーは「失敗」ではなく「次のヒント」です。**

# 第4章 Streamlit の基本を知る

## Streamlit とは何か

Streamlit は「PythonでWebアプリを簡単に作れる道具」です。HTML や JavaScript の知識は不要で、Pythonだけで作れます。

### 例: 1行で文字を表示

```python
import streamlit as st
st.title("私の最初のアプリ")
```

これだけで、ブラウザにタイトルが表示されます。

## 主な部品（ウィジェット）

Streamlit には、以下のような部品（ウィジェット）があります。

| 部品 | 用途 | コード例 |
|---|---|---|
| タイトル | 大見出し | `st.title("タイトル")` |
| テキスト | 説明文 | `st.write("説明です")` |
| 数値入力 | 数字を入力 | `st.number_input("年齢", 0, 120)` |
| スライダー | 範囲選択 | `st.slider("値", 0, 100, 50)` |
| プルダウン | 選択肢から選ぶ | `st.selectbox("病棟", ["5F", "6F"])` |
| ボタン | クリックで動作 | `st.button("実行")` |
| メトリック | 数値を大きく表示 | `st.metric("稼働率", "89%")` |
| グラフ | 折れ線・棒・円 | `st.line_chart(data)` |
| テーブル | 表形式 | `st.dataframe(df)` |

## 簡単な例: 稼働率計算アプリ

これを `app.py` に書いて `streamlit run app.py` するだけ：

```python
import streamlit as st

st.title("病床稼働率計算機")

# 入力
beds = st.number_input("総病床数", min_value=1, value=47)
patients = st.number_input("現在の在院患者数", min_value=0, value=42)

# 計算
occupancy = patients / beds * 100

# 表示
st.metric("稼働率", f"{occupancy:.1f}%")

# 判定
if occupancy >= 90:
    st.success("目標達成！")
elif occupancy >= 85:
    st.warning("あと少しで目標達成")
else:
    st.error("目標未達")
```

**これで、立派に動くアプリ**です。約15行、5分で作れます。

## ここまでできれば全てができる

実は、ベッドコントロールシミュレーターの巨大なコードも、**上記の部品を組み合わせているだけ**です。足し算が理解できれば九九もできるように、基本部品を理解すれば複雑なアプリも作れます。

# 第5章 GitHub にアップロードする

## なぜ GitHub を使うのか

### 理由1: 失敗しても戻せる
保存のたびに「セーブポイント」が作られ、いつでも過去に戻れます。

### 理由2: Streamlit Cloud にデプロイできる
GitHub にあるコードを Streamlit Cloud に読ませると、誰でもアクセスできるWebアプリになります。

### 理由3: 他の人と共有できる
同僚に「これを試してみて」と言う際、URLを送るだけでOK。

## GitHub の基本概念（3つだけ）

| 概念 | 意味 | 例え |
|---|---|---|
| **リポジトリ** | プロジェクト全体の箱 | ひとつのプロジェクトフォルダ |
| **コミット** | 変更を「保存」する単位 | Wordの上書き保存 |
| **プッシュ** | GitHubにアップロード | クラウドに送る |

## 最も簡単な方法: ブラウザでアップロード

ターミナル操作が苦手な方は、ブラウザだけで完結する方法があります。

### 手順

1. https://github.com にログイン
2. 右上の「+」→「New repository」
3. Repository name に好きな名前（例: `my-first-app`）
4. 「Public」を選択（無料で Streamlit Cloud に使える）
5. 「Add a README file」にチェック
6. 「Create repository」

これでリポジトリが作成されました。

### ファイルをアップロード

1. リポジトリのページで「Add file」→「Upload files」
2. `app.py` をドラッグ＆ドロップ
3. 下の「Commit changes」をクリック

これで完了です。

## requirements.txt を作る（重要）

Streamlit Cloud にアプリを動かしてもらうためには、「このアプリが使う道具のリスト」を渡す必要があります。それが `requirements.txt` です。

### 中身

```
streamlit
pandas
numpy
matplotlib
```

この4行だけのテキストファイルを作って、GitHubにアップロードします。

**これを忘れるとデプロイが失敗します。最重要ポイントです。**

# 第6章 Streamlit Cloud にデプロイする

## 手順

1. https://streamlit.io/cloud にログイン
2. 「New app」をクリック
3. リポジトリを選択（先ほど作ったもの）
4. Branch: `main`（デフォルト）
5. Main file path: `app.py`
6. 「Deploy」をクリック

**約1〜2分**で、アプリがインターネットで公開されます。

## 公開URL

デプロイが完了すると、以下のようなURLがもらえます：

```
https://your-app-name.streamlit.app
```

このURLを現場の同僚に送れば、誰でもブラウザでアクセスして使えます。**インストール不要、ログイン不要**です。

## 更新方法

アプリを改善したくなったら：

1. ローカルで `app.py` を編集
2. GitHub にアップロード（上書き）
3. Streamlit Cloud が **自動で** 再デプロイ
4. 1〜2分後、URLにアクセスすれば新しい版に更新済み

**これが「Continuous Deployment」という考え方です。難しい言葉ですが、要するに「コードを更新したら勝手に公開される」仕組みです。**

# 第7章 PDCAで改善していく

## 最初のバージョンを現場に出す

「完璧じゃないから出したくない」という気持ちをぐっとこらえて、**動くもの**ができたら早めに現場に出します。理由：

1. 現場の本当のニーズは、使ってもらって初めて分かる
2. 自分が想定していなかった使い方が発見される
3. 改善の優先順位が明確になる

## フィードバックの集め方

### 軽量な方法: 口頭で聞く

最初は口頭で十分です：
- 「使ってみてどうだった？」
- 「どこが使いにくかった？」
- 「何があったらもっと便利？」

### 中量: メモに集める

Apple Notes や Google Docs に以下をまとめる：

```
【フィードバック記録】

日付: 2026-04-10
提供者: 〇〇師長
良かった点:
- 稼働率が一目でわかる
改善要望:
- 月末予測も欲しい
- 病棟別に見られると嬉しい
```

### 本格的: GitHub Issues

GitHub にはIssueという機能があり、課題・要望を管理できます。個人で使う場合も便利です。

## 改善の優先順位

すべての要望に応えようとすると破綻します。**以下の順番**で優先順位をつけます：

1. **バグ修正**（計算が間違っている等）→ 最優先
2. **日常的に使う人が困っていること** → 高優先
3. **新機能追加** → 中優先
4. **見た目の改善** → 低優先

## 1週間サイクルで改善

**月〜金**: 現場で使う、フィードバック収集
**土日**: Claude と対話して1〜2個の改善を実装、GitHubにプッシュ
**月曜**: 新しい版で開始

このサイクルを続けると、1ヶ月で4サイクル、半年で24サイクルです。最初は些細だったアプリが、半年後には現場に欠かせないツールになります。

## 実例: ベッドコントロールシミュレーターの進化

| 時期 | 機能 | 行数 |
|---|---|---|
| v0.1 | 稼働率計算のみ | 20行 |
| v0.5 | 日次推移グラフ追加 | 200行 |
| v1.0 | A/B/C群管理、What-if分析 | 2,000行 |

# 第8章 AI（Claude）と上手に付き合う

## AIへの指示の基本

### 悪い例（曖昧）

```
いい感じに改善して
```

AI は何を改善すればいいか分からず、的外れな修正をします。

### 良い例（具体的）

```
日次推移タブに、月末の稼働率予測を表示する機能を追加したいです。

現在の表示:
- 直近20日の実績グラフ

追加したいもの:
- 現ペースが続いた場合の月末予測値
- 目標90%までの差（日数換算）

条件:
- 既存の計算ロジックは壊さない
- 画面の上部に目立つように表示
```

## AIに「やってほしくないこと」を伝える

AIは親切すぎて、聞いてもいないことまで修正しようとすることがあります。それを防ぐには：

```
以下の部分は変更しないでください:
- データ入力タブ（既に完成しているため）
- 運営分析タブの計算ロジック（別の担当が管理中）

今回の修正範囲:
- 日次推移タブの月末予測表示のみ
```

## エラーが出たときの伝え方

```
以下の操作をしたら、エラーが出ました。

操作:
1. 日次データ入力タブを開く
2. 日付を2026-04-10に選ぶ
3. 保存ボタンを押す

エラーメッセージ:
[エラー全文をここに貼り付け]

環境:
- Streamlit Cloud で動作中
- Python 3.11
```

## 一気に大きく変えない

AIは「500行の修正」を一度に出すこともできますが、**超初心者にとっては危険**です。

### 推奨: 小さく分けて進める

```
1. まず、入力欄を1つ追加してください。それを確認してから次に進みます
2. (動作確認後) 次に、その入力を計算に反映してください
3. (動作確認後) 最後に、結果をグラフに表示してください
```

1ステップずつ確認することで、どこで問題が起きたかが分かります。

## AIが間違うことを前提にする

AI は時々、間違ったコードを出します。特に：

- **数式の間違い**（小数点の扱い、単位の混同）
- **変数名のミス**（`患者数` と `total_patients` を混同）
- **ライブラリのバージョン違い**

**対策**: 必ず動かして結果を確認する。「AIが書いたから正しい」と思わない。

# 第9章 品質を保つ仕組み

## 本章で覚えること（1つだけ）

**数値の正しさを、アプリ本体だけに頼らない**

「動けばOK」ではなく、「正しく動いているか」を別の方法で確認する習慣をつけます。

## 方法1: 手計算で確認する

アプリが出した数値を、電卓で計算し直してみる。

例: アプリが「稼働率 89.4%」と表示した場合
- 電卓で: 42 ÷ 47 × 100 = 89.36…
- 小数点の丸めで 89.4% → 合っている ✅

## 方法2: 「スモークテスト」を作る

スモークテストとは、**基本的な計算が壊れていないかを自動でチェックする** 仕組みです。

### イメージ

```
確認したい計算:
1. 94床・在院80名 → 稼働率85.1% が返るか
2. 月150人入院 × LOS 18日 → 平均患者数90名 が返るか

毎回コード修正後に実行: 期待値と一致 → OK、不一致 → 警告
```

### 当院での実例

ベッドコントロールシミュレーターには `scripts/hooks/smoke_test.py` というファイルがあります。この中に、以下のような「期待値」が書かれています：

```python
EXPECTED = {
    "稼働率_全体": (89.0, 91.0),  # この範囲内なら正常
    "月平均在院日数_6F": (20.5, 22.5),
    ...
}
```

コードを修正したとき、このテストを実行します。期待値から外れたら「変な修正が混入したかも」と警告が出ます。

### 初心者はどうする？

まずは **エクセルで「答え合わせシート」を作る** だけでOK。

```
【答え合わせ】
入力: 病床94、在院80
期待する稼働率: 85.1%
アプリの結果: 85.1%  ← 一致 ✅

入力: 病床94、在院85
期待する稼働率: 90.4%
アプリの結果: 90.4%  ← 一致 ✅
```

修正後に、このシートを見ながら1つずつ確認します。

## 方法3: リグレッション防止

**リグレッション** = 「前は動いていた機能が、新しい修正のせいで壊れる」現象のこと。

超初心者が最も陥りやすい罠です。

### 対策

1. **バックアップを必ず残す** → GitHub を使っていれば自動でバックアップされる
2. **小さく修正する** → 1回の修正で複数の機能を同時に変えない
3. **修正前に「今動いている機能」のリストを作る** → 修正後に全部確認する

### チェックリスト例

```
【v3.0 リグレッションチェック】
- [ ] 稼働率計算 → 正しい数値？
- [ ] 日次推移グラフ → 正しく表示？
- [ ] フェーズ構成円グラフ → 表示される？
- [ ] What-if 分析 → 動く？
- [ ] データ保存 → エラー出ない？
```

修正のたびに全部チェック。面倒でもこれが命です。

# 第11章 よくある失敗とその対処

## 失敗1: 「完璧にしてから出そう」と半年止まる

**対処:** 「動くもの」ができたら、即現場に出す。不完全でもOK。使って初めて真のニーズが分かる。

## 失敗2: 「全機能を一気に作ろう」とする

**対処:** 最小限の機能（1〜2機能）から始める。後で増やせばいい。

## 失敗3: AI に丸投げして、出てきたコードを理解しない

**対処:** AI に「このコードを1行ずつ説明してください」と頼む。理解できるまで質問する。

## 失敗4: テストせずにデプロイ

**対処:** 必ず自分で動かして確認する。エクセルで答え合わせをする。

## 失敗5: 古い版を消してしまう

**対処:** GitHub にプッシュし続ける。ローカルだけに置かない。

## 失敗6: 一人で抱え込む

**対処:** 院内の誰か（IT好きな同僚、管理職）に相談する。ハッカソンや勉強会に参加する。

## 失敗7: 個人情報を扱ってしまう

**対処:** 患者名・ID・生年月日は絶対に入れない。集計値のみ。不安なときは院内の個人情報担当に確認する。

## 失敗8: 現場から反発される

**対処:** 「業務を奪うためのアプリではない」と明確に伝える。「あなたの負担を減らすためのもの」と強調する。実際に使ってもらって負担が減ったと感じてもらえれば、反発は消える。

## 失敗9: 長期間使わなくなる

**対処:** 毎日使う運用に組み込む（例: 朝会の前に必ず開く）。使わないと価値が出ない。

## 2026-04-10: 入力UIの統合と、Streamlit の意外な落とし穴

### 背景
2つの入力フォーム（日次集計と医師別詳細）が別々に存在し、同じ情報を
2回入力する必要があった。これを 1 つの統合フォームに集約する改修。

### 学び1: Streamlit の `st.form` は入力中にラベルが更新されない

**症状:**
「退院人数」を 0 → 5 に変更しても、各退院スライダーのラベルが
「退院1」〜「退院5」に切り替わらない。全部「(未使用)」のまま。

**原因:**
`st.form` で囲んだウィジェット同士は、送信ボタンを押すまで互いの変更を
認識しない仕様。これは Streamlit の公式な挙動。

**解決策:**
**フォームを廃止して通常ウィジェットにする** のが最もシンプル。これで
1つのウィジェットを操作するたびに Streamlit が自動再実行（rerun）し、
他のウィジェットも即座に反映される。

```python
# ❌ うまくいかない
with st.form("my_form"):
    count = st.number_input("人数", ...)
    for i in range(count):  # count がリアルタイム反映されない
        st.text_input(f"名前{i+1}")
    st.form_submit_button("送信")

# ✅ うまくいく
count = st.number_input("人数", ...)  # フォーム外
for i in range(count):  # count がリアルタイム反映される
    st.text_input(f"名前{i+1}")
if st.button("送信"):
    ...
```

**教訓:** フォームは「送信前にバリデーションを一括でしたい」ときだけ使う。
リアルタイム更新が必要ならフォームを使わない。

---

### 学び2: `clear_on_submit=True` は `key` 付きウィジェットをクリアしない

**症状:**
フォーム送信後、数値入力欄はリセットされたのにスライダーの値が前回の
ままになっている。

**原因:**
`st.form(clear_on_submit=True)` は value 引数で与えたデフォルト値には
戻すが、`key="xxx"` を明示したウィジェットは `st.session_state["xxx"]`
を保持するため、古い値が残り続ける。

**解決策:**
送信成功後、再描画前に明示的に `session_state` から該当キーを `pop` する。
ただし **ウィジェット作成後は書き換えられない** ため、リセットフラグを
session_state に立てて、次回再実行の冒頭で `pop` する。

```python
# 送信後
if submitted and is_valid:
    # ...データ保存...
    st.session_state["_reset_flag"] = True
    st.rerun()

# 次回再実行の冒頭（ウィジェット作成前）
if st.session_state.get("_reset_flag", False):
    for key in ["dm_slot_1", "dm_slot_2", ...]:
        st.session_state.pop(key, None)
    st.session_state["_reset_flag"] = False
```

**教訓:** Streamlit の widget lifecycle は独特。`key` を使ったら
クリーンアップの責任は自分にある。

---

### 学び3: Streamlit Cloud のデプロイが反映されない時の対処

**症状:**
GitHub には push 済みで、コードも正しい。でも `xxx.streamlit.app` を
開いても古いバージョンが表示される。

**原因（複数の可能性）:**
1. Streamlit Cloud のビルドキュー待ち（1-3分程度）
2. ブラウザのキャッシュ
3. Streamlit Cloud 側でアプリが古いプロセスのまま固まっている

**対処フロー（上から順に試す）:**

1. **1〜3分待つ** — 自動デプロイには時間がかかる
2. **ハードリフレッシュ** — Mac は `Cmd + Shift + R`
3. **ブラウザを変える** — Safari で動かなくても Chrome なら動くことがある
4. **Streamlit Cloud の管理メニューから Reboot app**
   - アプリ画面の右上「⋮」メニュー → 「Reboot app」
   - メニューが見つからない場合は `share.streamlit.io` にログインして
     アプリ一覧から直接 Reboot
5. **空コミットで再デプロイをトリガー**
   ```bash
   git commit --allow-empty -m "chore: 再デプロイトリガー"
   git push
   ```

**教訓:** 「コードは正しいのに反映されない」は Streamlit Cloud あるある。
慌てずにキャッシュ・Reboot・空コミットの順で試せばほぼ解決する。

**デバッグのコツ:** 目印となる小さな変更（例: `st.caption("🆕 v3")`）を
追加して push すると、反映されたかどうかが一目でわかる。

---

### 学び4: 「フォーム」か「通常ウィジェット」かはユーザー体験で決める

当初はフォームで作り始めたが、リアルタイム更新を求めるならフォームを
外す方が明らかに使いやすい。**「入力が 6 分かかってもいいから 1 回で終わる」**
というユーザーの言葉が決め手になった。

**教訓:**
- 技術的に複雑な解決策（reset flag、pending flag など）を重ねる前に、
  **「そもそもフォームいるのか？」** と問い直す
- ユーザー体験（マウス操作だけで完結する等）を設計の起点に据える
- 「入力回数を減らす」より「**二度手間をなくす**」の方が価値が大きい

### 学び5: 関数定義の順序は「使用箇所より前」が鉄則

**症状:**
実績データモードに切り替えたら `NameError: _filter_current_month is not defined`。

**原因:**
関数 `_filter_current_month` がモジュール内で line 1555 に定義されていたが、
line 1335 で呼び出されていた。Streamlit はモジュールを毎回上から下に
実行するため、呼び出し時点ではまだ関数が定義されていない。

関数内からの呼び出し（遅延バインディング）であればこの順序は問題ないが、
**モジュールトップレベルからの呼び出し** では順序が重要。

**解決策:**
関数定義を最初の使用箇所より前に移動する。

**教訓:**
- モジュール直下（関数内ではない場所）で関数を呼ぶときは、**定義が先**
- こういう潜在バグは「普段使わない処理パス」に隠れている（この例では
  実績データモード）。**各モードを定期的に触って確認する** クセをつける

### 学び6: 同じ日付の 5F と 6F を区別できる UI にする

**症状:**
記録データ一覧に「2026-04-10」の行が 2 つあって、どっちが 5F でどっちが
6F かわからない。削除ドロップダウンも日付だけで区別がつかない。

**原因:**
テーブルの表示カラムに `ward`（病棟）列が含まれていなかった。
削除ドロップダウンも日付文字列のみを使っていた。

**解決策:**
- テーブルの日付の隣に「病棟」列を常時表示
- 削除ドロップダウンを `"2026-04-10 (5F)"` のような複合ラベルに
- `delete_record()` に ward 引数を追加（デフォルト None で後方互換性維持）

**教訓:**
- **複合主キー**（日付+病棟など）を扱うなら、UI でも複合で見せる
- 後方互換性を意識した引数追加（デフォルト値）は、既存コードを壊さない

### 学び7: 「増える」方向の発見は経営インパクトが大きい

Phase 2 の実装中、短手3 除外後の平均在院日数が **減る** と思っていたら、
実は **増える** ことが判明した（短手3 は超短期入院なので、除外すると
平均が長くなる方向に動く）。

| | 通常 | 短手3除外後 |
|---|---|---|
| 5F 3ヶ月 rolling LOS | 18.7日 | **23.0日** (+4.3日) |

これは 2026 年改定で施設基準判定が **厳しくなる** ことを意味する。
実装のテストを書いていなければ気づかなかったかもしれない。

**教訓:**
- **計算の方向性を直感で決めない**。必ず実データでテストする
- テストの assertion は「期待値を書く」だけでなく、「**期待の符号**」
  （増えるのか減るのか）が正しいかも確認する
- こういう発見は**そのまま経営判断の材料になる**。リサーチ文書に追記する
