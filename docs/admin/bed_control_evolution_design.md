# ベッドコントロールシミュレーター 進化設計書

## 作成日: 2026-04-04
## ステータス: 設計確定 → 実装着手

---

## 1. ビジョン

**精神論を、数字に変える。**

より良い病棟稼働貢献のために、持続的な病院運営のために、こうすればより良くなるという的確な指摘を見える化する。自分からその改善に働きかけたくなるきっかけを得る。なぜなら、それによって病院が発展する余力となる資源が増え、働く現場の賞与が増えモチベーション維持に繋がるから。

## 2. 解決すべき問題

### 2.1 現状の課題

| 課題 | 現状 | 目指す姿 |
|------|------|----------|
| 稼働率1%の価値が共有されていない | 精神論で「上げよう」としか言えない | 「1% = 年間1,113万円 = 営業利益の31%相当」と数字で示す |
| 退院日の決定権 | 一部医師が圧力で退院日を決定（取り決め違反） | 曜日偏在の影響を金額で可視化し、対話の根拠を提供 |
| 入院創出の努力が見えない | 外来医師が入院に繋いでも記録されない | 入院創出医を記録し、貢献を可視化 |
| 医師別の行動差が不可視 | 誰でも気づいているが「どうしようもない」 | データで差を示し、「放置できない」に変える |
| 改善の効果が実感できない | 改善しても何に繋がるかわからない | 金額換算・営業利益比率で示す |

### 2.2 具体的な現場の問題

- 退院許可を一方的に出す医師（週末に患者を減らしたい動機）
- 入院にすべき患者を外来で返している医師
- 学会前に意図的に入院を減らす医師
- 専門外の救急を断る医師
- すぐ高次病院へ転院させる医師
- 非常勤医師からの入院発生が少ない
- 高次病院からの下り搬送が閑散期に減る

## 3. 経営基盤データ（令和7年度実績）

| 項目 | 数値 |
|------|------|
| 総売上高 | 30億7,890万円 |
| 人件費率 | 58% |
| 営業利益 | 3,550万円 |
| 営業利益率 | 約1.15% |
| 地域包括医療病棟 病床数 | 94床 |
| 地域包括医療病棟 平均稼働率 | 89% |
| 1床1日あたり入院単価 | 30,500円（2026年6月〜: 32,000円） |
| 職員数 | 290人 |

### 3.1 稼働率1%の価値

- 94床 × 1% = 0.94床
- 0.94床 × 365日 × 30,500円 = **年間約1,113万円**
- 営業利益3,550万円に対して **31%相当**
- 人件費率58%で換算すると、一人あたり **年間約2.2万円**

## 4. 機能設計

### 4.1 常時表示の基盤指標

画面上部またはサイドバーに常時表示：

```
現在の稼働率 89% ｜ あと1%で年間 +1,113万円（営業利益の31%相当）
```

- どのタブを見ていても常に目に入る位置
- 稼働率の変化に応じてリアルタイム更新
- 2026年6月以降は入院単価32,000円で再計算

### 4.2 「改善のヒント」機能

#### 概要
アプリが運用データを分析し、改善の種を自動検出して提示する機能。

#### 表示イメージ
```
⚠️ 金曜退院の集中
先月の金曜退院は32件（全体の40%）。土日の平均稼働率が82%に低下。
年間推定ロス: 576万円（営業利益の16%相当）
→ シミュレーションを見る

💡 A先生の入院創出が増加
先月の入院創出: 12件（前月比+4件）。稼働率への貢献: +0.3%
→ 詳細を見る
```

#### 設計思想
1. **何が起きているか** — 事実の提示
2. **それがいくらの影響か** — 金額換算
3. **営業利益に対してどれほどか** — 比率で実感させる
4. **どうすれば改善するか** — シミュレーションへの導線

#### 体験の3ステップ
1. **気づき** → 改善のヒントが問題を自動提示
2. **確認** → データで裏付けを自分の目で確かめる
3. **対話** → 根拠を持って相手に示す（画面提示・PDF・院内メール）

#### 検出するヒントの種類（初期）
- 退院の曜日偏在（金曜集中など）
- 空床ラグ（退院から次の入院までの日数）
- 医師別の入院件数の偏り
- 医師別の退院曜日パターン
- 医師別のフェーズ構成の偏り
- 入院経路別の件数推移
- 稼働率トレンドと目標との乖離
- 改善を複数積み重ねた場合の合計効果

### 4.3 医師マスター設定機能

#### 医師情報の管理画面
- 医師名の追加・編集・削除
- 属性の設定（以下から選択）：
  - 常勤病棟担当医師
  - 常勤外来のみ担当医師
  - 非常勤医師
  - 常勤救急応援医師
- 「連携室」等の部署も入院創出元として登録可能
- 入院経路の管理：
  - 外来紹介
  - 救急
  - 連携室
  - ウォークイン
  - （編集可能）

### 4.4 日次データ入力の拡張

#### 入院時の入力（3ステップ）
1. **経路を選ぶ**（プルダウン）— 外来紹介 / 救急 / 連携室 / ウォークイン
2. **入院創出医を選ぶ**（プルダウン）— その入院を生み出した医師（いれば）
3. **入院担当医（主治医）を選ぶ**（プルダウン）— 病棟で受け持つ医師

※ 入院創出医と入院担当医は同じ場合も異なる場合もある
※ 経路が連携室・ウォークインなど医師不在の場合、入院創出医は省略可

#### 退院時の入力（2ステップ）
1. **担当医を選ぶ**（プルダウン）
2. **在院日数を入力**（数字）

※ 在院日数からフェーズ（A/B/C）は自動分類
※ 将来の退院経路拡張に備え、拡張可能な設計とする

### 4.5 医師別分析タブ（新規）

日次入力データの蓄積により、以下を自動集計・可視化：

- 医師別の月間入院件数
- 医師別の月間退院件数
- 医師別の退院曜日分布
- 医師別の在院日数分布
- 医師別のフェーズ（A/B/C）構成
- 医師別の稼働貢献度（金額換算）
- 入院経路別の件数推移
- 入院創出医 vs 入院担当医の対応関係

## 5. 将来構想（第2層・第3層）

### 第2層: 改善仮説の保存・比較
- What-if分析を「名前付き仮説」として保存
- 複数仮説の同時シミュレーション
- 改善の積み重ね効果の可視化

### 第3層: 提案書ドラフト自動生成
- 検証済み仮説を経営会議向けフォーマットに変換
- 現状→目標→想定効果→施策→根拠データの構造
- PDF・院内メールでの配布

### 立場別ビュー（将来）
- **コントローラー向け**: 改善アクションの優先順位、全体ダッシュボード
- **医師向け**: 自分の入退院パターンと全体への影響
- **看護師向け**: 改善の積み重ねが人件費原資としていくらになるか
- **経営者向け**: 経営指標サマリーと改善トレンド

## 6. 実装優先順位

1. **医師マスター設定機能** — 他の機能の前提
2. **日次データ入力の拡張** — 経路・入院創出医・担当医・在院日数
3. **医師別分析タブ** — データが溜まれば自動で機能
4. **常時表示の基盤指標** — 稼働率1%=年間○万円
5. **「改善のヒント」機能** — データ蓄積後に最大効果を発揮

## 7. 技術的考慮事項

- 電子カルテとの連携なし（コストゼロ運用）
- 入力はコントローラーの手入力
- 入力の省力化が最重要（プルダウン選択中心）
- 医師マスターは設定画面で編集可能に
- 既存のbed_data_manager.pyのデータ構造を拡張
- Streamlitのsession_stateとCSV保存の既存パターンを踏襲
