# 多職種退院調整・連休対策カンファ 運用台本 v4

**作成日:** 2026-04-19
**対象アプリ:** ベッドコントロールシミュレーター（`scripts/bed_control_simulator_app.py`）
**画面:** サイドバー「🏥 退院調整」→「🏥 カンファ資料」タブ
**想定参加者:** 病棟師長・主任看護師・リハ PT/OT/ST・退院支援看護師（MSW）・担当医・副院長（任意）
**想定所要時間:** 20〜30 分（通常モード）、40〜60 分（連休対策モード）
**想定場面:** 病棟詰め所の据置 PC の前に 4〜6 名が立って／座って画面を囲む

---

## この台本の使い方

- 「何を画面で見て」「誰が発言して」「誰が操作して」「何が記録される」を時系列で書いています
- 参加者は台本を読まずに済むよう、司会（通常は師長）だけがこのファイルを開いておけば回せます
- 病棟ごとに運用が違う場合は、自院の呼び方・時間帯に合わせて章ごとの文言を調整してください
- 「操作：」の行はアプリの操作、「発言例：」の行は口頭でのやり取り、「画面上の変化：」の行は UI 側の反応を示します
- v4 では新機能として **📉 過去実績分析タブ** への「寄り道」（第6章）と **患者ステータス履歴の差分表示** が加わっています

---

## このカンファの 4 つの原則

1. **一画面で完結する** — 画面スクロールなしで 16:9 に収まるよう設計されている。スクリーンショット 1 枚で会議録代わりになる
2. **誰でも操作できる** — ステータスタグをクリックするだけでよく、PC 操作に慣れていないメンバーでも 30 秒で覚えられる
3. **記録は自動で残る** — ステータス変更は即座に JSON atomic write で保存され、次回カンファで前回の続きから差分表示される
4. **個人情報は現場内で完結する** — 患者名・ID の編集フィールドは任意で、入力しても Git には上がらず院内 PC にのみ保存される

## 第1章：Block A — 本日のサマリー（KPI 共有、2 分）

### 見る場所：画面中央上部の KPI 横並び（Block A）

画面には 5 つの指標が並んで表示される：

| 位置 | 指標 | 何を意味するか |
|------|------|--------------|
| 1 | 今月の稼働率 | 今月の実績 vs 目標（全病棟 一律 90%） |
| 2 | 平均在院日数（3 ヶ月 rolling） | 制度上限 21 日が基準、推定 LOS 5F 16.5 日 / 6F 15.6 日 |
| 3 | 救急搬送後 15%（rolling 3 ヶ月） | 地域包括医療病棟の施設基準。参考: 5F 実績 53.1% / 6F 61.1% |
| 4 | フェーズ構成 | 導入期 / 治療期 / 退院調整期の人数と比率 |
| 5 | 退院候補 | 今週中に退院可能な患者数（Block C と連動） |

**補足:** 2026-06-01 以降は救急搬送後 15% が本則適用（rolling 3 ヶ月・病棟別）で運用されます。

### 発言例（司会）

> 「今月の稼働率は 87%、目標 90% に対してマイナス 3 ポイント。ALOS は 16.8 日で制度上限 21 日までまだ余裕があります。救急搬送後は rolling 3 ヶ月で 5F 54%、基準達成。フェーズ構成は導入期 8 名・治療期 24 名・退院調整期 15 名。今週の退院候補は 6 名と出ています」 <!-- qa-skip -->

### なぜこの共有が必要か

- 「稼働率が低い」だけでは対策が決まらない
- 「稼働率 87%・ALOS 余裕・救急達成」なら、**延ばすより新しい患者を受け入れる** 方針 <!-- qa-skip -->
- 「稼働率高い・ALOS 21 日近い・救急ぎりぎり」なら、**連休前退院を加速** する方針
- この 2 分の状況確認で、今日のカンファのトーンが決まる

## 第2章：Block B — 今週の見通し（需要予測、3 分）

### 見る場所：画面中央左側（Block B）

Block B に表示されるもの：

- 通常モード：金・土・日の 3 日分の空床予測と救急余力（棒グラフ + 数値）
- 連休対策モード：連休 5 日分（例：4/29 水〜5/3 日）の空床予測と救急余力
- 下段に「週末受入余力 XX 床日（応需可能だった患者機会）」
- さらに下にステータス内訳（Block C で更新される値が自動集計される）

### 発言例（司会）

**通常モード時：**
> 「5F は今週末、金 4 床・土 8 床・日 7 床の空床見込み。土日の救急余力がマイナス 1、ほぼ埋まる前提です。応需可能だった患者機会は合計 19 床日」

**連休対策モード時：**
> 「GW 5 日間で 5F の空床見込み合計 66 床日。連休中は入院がほぼ止まるので、連休前にどれだけ退院できるかがこの 66 床日を何床日まで減らせるかに直結します」

### 用語の意図

- **受入余力**という言葉を使います。「空床コスト」「損失」という経営用語は避け、「受け入れられた可能性のある患者数」として可視化しています
- **応需可能だった患者機会**は、医療倫理上の観点から「空いていた床を金額で語らない」ための表現です。理事会向けでは運営貢献額・空床の影響額として併記しますが、カンファ現場では床日ベースで話します

## 第3章：Block C — 患者一覧とステータス更新（10〜15 分・本番）

### 見る場所：画面中央右側（Block C・ここが議論の中心）

画面に最大 10 名の患者行が表示される。在院日数の降順。各行の構成：

| 列 | 内容 | 備考 |
|---|------|------|
| 医師 / 患者 | 主治医の姓 + 患者名・ID（任意編集） | 患者名・ID は空欄でも運用可 |
| 病棟 | 5F / 6F | 色分け表示 |
| 在院 | Day N | 在院日数。85 歳以上は行右端に🎂バッジ（当院実績 27.3%） |
| 退院予定 | 日付または「未定」 | サンプル値。編集は別機能 |
| 確認事項 | 短文（リハ・MSW・主治医へのメモ） | 事前に入力済み |
| ステータス | クリッカブルタグ（現在の分類） | ★ ここがメイン操作 |
| 📜 履歴 / ✏️ 編集 | 横並び 2 ボタン | 2026-04-18 修正済、同列で操作 |

### 7 つのステータス（通常モード）

| タグ | 絵文字 | 意味 | 誰が発言のトリガーか |
|------|-------|------|-------------------|
| 方向性未決 | ⚫ | 退院先も退院時期も決まっていない | 医師または MSW |
| 医学的 OK 待ち | 🔵 | 主治医の退院判定が残っているだけ | 主治医 |
| 家族希望待ち | 🟢 | 医学的には退院可、家族の迎え日待ち | MSW・病棟看護師 |
| 施設待ち | 🟡 | 老健・特養・療養病床の受入待ち | MSW |
| 介護保険待ち | 🟣 | 認定調査・担当者会議・ケアマネ調整中 | MSW |
| リハ最適化中 | 🟠 | 機能目標に向けてリハ継続中 | リハ PT/OT/ST |
| 新規 | 🆕 | 初回カテゴライズ前 | 全員 |

### 5 つのステータス（連休対策モード、v4 分散退院型）

| タグ | 絵文字 | 意味 |
|------|-------|------|
| 連休前退院可 | ✅ | 連休前日までに退院、家族が平日お迎え OK |
| 連休中退院可（Day 2-3） | 🎌 | 連休中に退院、**休日の家族お迎えを活用** |
| 連休明け退院可 | 📅 | 連休後最初の平日に退院 |
| 連休中継続ケア | 🏥 | 連休中病院で過ごす方針確定 |
| 調整中 / 新規 | 🆕 | 家族相談中 or 未カテゴライズ |

### 📜 履歴ボタン（v4 強化）— 前回からの変化を確認

**操作：**
1. 患者行の「📜 履歴」ボタンをクリック
2. ポップオーバーが開き、その患者のステータス履歴が時系列で表示される

**画面上の変化：**
- 「2026-04-12 (前回カンファ): 🟠 リハ最適化中」
- 「2026-04-19 (今回): ?」と現時点のエントリが空欄で表示される
- 前回からの経過日数（Day +7 など）も併記される

**発言例（リハ PT）：**
> 「この 80 代女性、前回 🟠 リハ最適化中だったけど、歩行 100m 到達して退院時機能目標クリアしたので、今週 🔵 医学的 OK 待ちに上げていいです」

### カテゴリ分けの判断基準（ここが議論の肝）

- **⚫ 方向性未決 vs 🔵 医学的 OK 待ち** の違い
  - 「主治医の OK が出れば退院先が見える」なら 🔵
  - 「退院先候補すら絞れていない」なら ⚫
  - 30 秒迷ったら ⚫ に寄せる（次週再議論でよい）

- **🟢 家族希望待ち vs 🟡 施設待ち**
  - 自宅退院予定 → 🟢
  - 施設転院予定 → 🟡
  - 決まっていない → ⚫

- **🟠 リハ最適化中** は「退院可能性はあるが、もう少しリハで ADL を上げた方が長期予後に有利」と判断した場合のみ
  - 単純な延長希望ではない。「退院時機能を 1 点上げると 1 年死亡リスクが下がる」というエビデンスに基づく判断

### 操作の流れ（参加者全員で）

**発言例（師長）：**
> 「じゃあ上から順に見ていきます。伊藤医師の患者 Day 35、退院予定未定。4/24 までに退院目処を再評価、要再カンファとなっています。今日のカテゴリどうしましょう」

**発言例（MSW）：**
> 「退院先、まだ候補出せていません。方向性未決ですね」

**操作（師長が PC で）：**
1. その患者行の「ステータス」列のタグをクリック
2. ポップオーバーが開く
3. ラジオボタンで「⚫ 方向性未決」を選択
4. ポップオーバー外をクリックするか、そのまま次のステータスへ進む

**画面上の変化：**
- タグの色・絵文字が即座に ⚫ 方向性未決 の黒タグに変わる
- 画面左の「在院患者ステータス内訳」の「⚫ 方向性未決」カウンタが +1 される
- 裏側では `data/patient_status.json` に atomic write で自動保存される（次回起動時も復元される）
- 履歴にも「2026-04-19: ⚫ 方向性未決」が追加される

### 編集ボタン（✏️）— 患者名・ID の追加入力（任意）

**操作：**
1. 患者行の「✏️」ボタン（履歴ボタンと横並び）をクリック
2. ポップオーバー内に「主治医名」「患者名」「患者 ID」のテキストフィールド
3. 入力してフォーカスを外すと即保存

**運用ルール（厳守）：**
- 個人情報は **現場 PC 内で完結**（`data/patient_names.json` は `.gitignore` で除外）
- クラウド同期対象外、院内 PC にのみ残る
- スクリーンショット共有時は事前に「🗑 データ管理（一括クリア）」で氏名・ID をクリア
- カンファ中だけ一時入力し、終了時にクリアする運用も可

### 10 名全員のカテゴリ分けが終わったら

**発言例（師長）：**
> 「⚫ 未決 3 名、🟡 施設待ち 2 名、🟠 リハ最適化中 2 名、🔵 医学的 OK 待ち 1 名、🟢 家族希望待ち 1 名、🆕 新規 1 名。この未決 3 名が来週のキーですね」

## 第5章：エビデンスバー（ローテーション 12 件・1 分）

### 見る場所：画面最下部（紺色のエビデンスバー）

1 行のエビデンスが毎カンファで 1 件ずつローテーション表示される。
2026-04-18 に全 80 件を PubMed 一次文献として精密検証済み（ハルシネーション 0、PMID・DOI 100% 一致）。

**表示例：**
> **エビデンス** 87 歳超でも多要素運動で Barthel が対照群 -5.0 → 介入群 +1.9 へ逆転 — 超高齢でも機能改善は可能 　Martínez-Velilla 2019 | JAMA Intern Med | PMID: 30419096

### 運用

- プール 80 件のうち、毎日表示されるのは **ローテーション 12 件**（`rotation_eligible: true`）
- 残り 68 件は「折りたたみ・勉強用」で、学習会で深掘りする
- テーマは「入院延長 × リハ介入 × 高齢者予後改善」に限定
- 印刷時には自動非表示（`@media print { display:none }`）で紙面を汚さない
- カンファ冒頭または終盤で司会が 15 秒読み上げる運用推奨
- 深追いしない。月 1 回気になった 1 件を学習会のきっかけにする

### エビデンスの狙い

「退院日の決定は、ベースライン ADL 復帰ではなく退院後の予後を見据えた機能最適化で判断する」というパラダイムを、カンファの中で少しずつ浸透させるための仕組みです。押しつけは避け、数字で静かに示します。

## 第6章：過去実績タブへの寄り道（v4 新機能、2 分）

### いつ使うか

カンファ中に議論が詰まった瞬間、特に次のような質問が出たときに「📉 過去実績分析タブ」へ 1 クリックで寄り道する運用です：

- 「この退院判断、例年と比べて早いのか遅いのか」
- 「本日の退院ラッシュ（退院 5 名）は異常なのか、金曜だと普通なのか」
- 「今月の入院ペースは例年より速い？遅い？」
- 「予約 → 入院までのリードタイムはどれくらい見込めば良いか」

### 操作の流れ

**操作（師長が PC で）：**
1. サイドバーから「📊 今日の運営」をクリック（カンファ資料タブから一時離脱）
2. 右側タブから「📉 過去実績分析」をクリック（2025FY 1965 件の実データから集計）
3. 確認後、サイドバー「🏥 退院調整 → 🏥 カンファ資料」に戻る

**所要時間：** 30 秒〜1 分の寄り道（カンファを止めない）

### 寄り道ケース 1：月別推移で今月の入院ペースを比較

**発言例（副院長）：**
> 「今月の入院ペースが例年より速い気がするんだけど、過去実績タブで確認しましょう」

**操作・画面上の変化：**
- 月別推移グラフで 2025 年 4 月の入院数と 2026 年 4 月（当月）の累積入院数を重ねて表示
- 「4/19 時点で当月 87 件、例年同時期 75 件、+16% 速い」と数値要約

**判断：**
- 「例年より速いなら、退院を少し前倒ししないと満床リスクが高まる。Block C の 🟠 リハ最適化中 2 名を 🔵 医学的 OK 待ちに繰り上げるか議論しましょう」

### 寄り道ケース 2：曜日パターンで今日の退院ラッシュをチェック

**発言例（師長）：**
> 「今日退院 5 名予定だけど、これって金曜として普通？」

**操作・画面上の変化：**
- 曜日別退院数のヒストグラム表示
- 「金曜の退院中央値 4 名、今日 5 名は +1 名で +25%、常識的な範囲」

**判断：**
- 「金曜としては少し多めだけど異常値ではない。予定通り進めて良さそう」

### 寄り道ケース 3：リードタイム中央値で入院見込みを調整

**発言例（MSW）：**
> 「連休前の空床を埋めるために、予約患者の入院前倒しは可能でしょうか」

**操作・画面上の変化：**
- 「予約 → 入院」リードタイム分布グラフ
- 「中央値 12 日、IQR 8-17 日。前倒し打診は 7 日前が現実的下限」

**判断：**
- 「7 日前倒しは可能性あり。外来主治医に打診リスト作ります」

### この章の狙い

- カンファ中の「なんとなく多い気がする」「例年こんなもん？」という感覚論を、30 秒で **数字に置き換える**
- 議論を止めず、判断材料だけを素早く足して戻る
- 過去実績 1965 件は救急率 5F 53.1% / 6F 61.1%、85 歳以上 27.3% 等の実データで、当院の「普通」を定量的に知る根拠になる

## 第7章：通常モード ⇔ 連休対策モード切替（v4 分散退院型）

### v4 の運用思想転換（副院長 2026-04-19 指示）

v4 では連休対策の発想が **「連休前集中型」から「連休前 + 連休中の分散退院型」** に転換しています。

**従来の課題:**
- 連休前に退院を集中 → 病棟スタッフの負荷が前倒しで集中
- 連休中の稼働率が 60% 台まで落ち込み、診療報酬が大幅低下 <!-- qa-skip -->（従来型の問題イメージ）
- 家族が平日に仕事を休んでお迎えに来る必要がある

**v4 分散型の 3 本柱:**

1. **連休中退院の推奨化** — 従来の「日曜退院推奨」と同じロジックを連休中にも拡張。連休 5 日間なら、連休前だけでなく Day 1-3 にも退院を配置
2. **一日 N 名までの退院枠** — 連休 5 日で合計 M 名を分散（一日に集中させない）
3. **休日の家族都合を逆に利用** — 平日お迎えより休日お迎えの方が家族にも優しい

**効果イメージ:** 連休中稼働率 60% → 85% に改善、連休 1 回あたり数百万円規模の診療報酬改善。 <!-- qa-skip -->（前後比較イメージ）

### 切替の仕組み — 手動切替 + 推奨バナー

- 連休対策モードは **師長が手動で ON** します（自動切替ではありません）
- 連休まで **21 日以下** になると、画面上部に推奨バナー「💡 連休まで N 日 — 連休対策モードへの切替を推奨します」が自動表示されます
- **14 日を切ると警告色（橙）、7 日以内は危険色（赤）** で強調表示されます
- バナーを見て判断し、**サイドバーの「連休対策モード」トグルを ON** にします
- 連休明け後、師長が手動で OFF に戻します

### 切替手順

1. 画面上部の推奨バナー（💡 連休まで N 日…）を確認
2. サイドバーの「連休対策モード」トグルを ON
3. 画面上の変化を確認：
   - モードバッジが「🚨 連休対策モード」赤に変わる
   - 推奨バナーは非表示になる（切替済みのため）
   - Block B が連休 5 日分の見通しに切り替わる
   - Block C のステータス候補が 7 種 → **5 種**（連休前退院可 / **連休中退院可（Day 2-3）** / 連休明け退院可 / 調整中 / 新規）に変わる
   - Block D の職種別アクションも分散型向け文言に変わる
   - 所要時間目安が 20〜30 分 → 40〜60 分に伸びる

### 通常モード vs 連休対策モードの使い分け

| 項目 | 通常モード | 連休対策モード（v4 分散型） |
|------|---------|-------------|
| 想定頻度 | 毎週木曜 | 大型連休 7 日前〜前日 |
| 所要時間 | 20〜30 分 | 40〜60 分 |
| 主焦点 | Block C 10 名のカテゴリ分け | **連休前 + 連休中の分散退院計画** |
| ステータス数 | 7 種 | **5 種（連休前 / 連休中 / 連休明け + 調整中 / 新規）** |
| Block B 表示 | 金・土・日 3 日分 | 連休 5 日分 |

### 切替後の 3 週間の進め方（v4 分散型）

| 週 | 目標 |
|----|------|
| 連休 3 週前 | 対象患者の洗い出し、3 タグ（連休前 / 連休中 / 連休明け）の初回分類 |
| 連休 2 週前 | **家族への事前説明**「連休中のお迎えはご都合いかがですか？」、調整中 → 確定への移行 |
| 連休 1 週前 | **一日 N 名の退院枠を確定**、搬送・迎え時間の最終確認、継続患者の連休中指示書確認 |

### 家族への事前説明トーク例（担当看護師・医師用）

カンファで「連休中退院可（Day 2-3）」タグに分類したら、家族に以下のように切り出します:

「○○さんの状態が安定していれば、**連休中のお迎えはご都合いかがですか？** 平日にご家族が仕事を休んで来ていただく必要がなく、ご家族にもご負担が少ないと思いまして」

家族が休みである休日は、かえってお迎え調整がしやすいケースが多いことが運用上わかっています。

### 一日 N 名までの退院枠設定（カンファで合意）

**決め方:**
- 連休前日の空床見込み + 救急受入必要数 + スタッフ人数 → 一日の退院可能枠 N を逆算
- 連休 5 日間で合計 M 名を分散（例: Day 1=3 名、Day 2=2 名、Day 3=2 名、Day 4=1 名、Day 5=0 名）
- What-if タブで「連休中 Day 2 に X 名退院させると稼働率はどう変わるか」をシミュレーションして合意形成

### 連休明けの再切替

- 連休明けの火曜カンファで、師長が **手動で** 通常モードに戻す（自動復帰はしません）
- データは保持されるが、カテゴリ体系が変わるので全患者が 🆕 新規 になる
- 連休中退院・連休前退院で不在になった患者は、次回カンファ時に枠外になる

## カンファ終了・記録（2 分）

### 記録の仕組み

- ステータス変更・患者情報編集はすべて即時 JSON atomic write で保存されているので、「保存を忘れた」は起きない
- 履歴 JSON により、次回カンファの冒頭で自動的に「前回からの変化差分」が表示される
- ただしセッションが閉じるとブラウザ内のタブ状態は消えるので、**画面全体のスクリーンショットを撮影する** 運用にする
- スクリーンショットは病棟 PC のフォルダ（院内 LAN で師長・MSW と共有）に保存

### 推奨：スクリーンショット印刷

- ブラウザの「印刷」機能で PDF 化 → 院内プリンタで A3 横印刷
- 印刷時は自動的にエビデンスバーと編集ボタンが消えて、カンファ結果だけが紙に残る
- 医療記録として保存する場合は患者名・ID を伏せ字にするか、編集欄を空欄のまま運用する

### 次回カンファへの引き継ぎ

- ステータスと履歴は JSON 保存されているので、1 週間後に同じ画面を開くと **前回のステータスから再開・差分バッジ表示** される
- 新規入院が発生している場合は、サンプルデータ 10 名の枠内でカテゴリを 🆕 に戻す運用

### 発言例（師長・カンファ閉会）

> 「今日のカンファはここまで。スクリーンショットを撮って 6F 師長にも共有します。来週の木曜日、同じ時間に再集合でお願いします」

## Appendix C：トラブルシューティング

| つまずき | 原因 | 対処 |
|---------|------|------|
| カンファ中にステータスタグが反応しない | ブラウザが固まっている | ページを再読み込み（ステータスは atomic write 済） |
| 前回のステータスが残っていない | データ管理で一括クリアした後 | `data/patient_status.json` は最後の保存日時を確認 |
| 連休対策モードに切り替えたのに 7 タグのまま | モードトグル切替直後はリセットされる仕様 | 一度全患者 🆕 に戻り、そこから 4 タグに分類し直す |
| スクリーンショットに患者名が写る | 事前に名前を入力してあった | 印刷前に「🗑 データ管理（一括クリア）」を実行 |
| 画面が 16:9 に収まらない | ブラウザのズームが 125% 以外 | Ctrl/Cmd で 125% に合わせる（v3.6 以降の推奨倍率） |
| エビデンスが毎回同じ | 乱数シードの問題 | アプリ再起動で別ファクトが出る |
| 📉 過去実績分析タブが見つからない | サイドバーセクションが違う | 「🏥 退院調整」ではなく「📊 今日の運営」セクションにある |
| 履歴が空で表示される | 初回カンファなので過去データがない | 2 回目以降から差分バッジが出る |
