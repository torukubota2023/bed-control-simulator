自動化されたスモークテスト（`scripts/hooks/smoke_test.py`）と合わせて使用する。

## 前提条件

- `streamlit run scripts/bed_control_simulator_app.py` でローカル起動

## 手動チェック（UIで確認）

### A. 表示モード × 病棟選択の組み合わせ
- [ ] 実績モード × 全体 → 稼働率88.4%、Δ +4.8pt 表示
- [ ] 実績モード × 5F → 稼働率86.3%、単体目標97%表示、フェーズ構成タブで C群が0%でない
- [ ] 実績モード × 6F → 稼働率90.5%、単体目標89%表示、rolling LOS 22.6日（基準超過アラート）
- [ ] シミュレーションモード × 全体 → NameError なし
- [ ] シミュレーションモード × 5F/6F → 同上

### B. 日次推移タブ
- [ ] 4月1日〜20日のみ表示（過去90日データが混入していない）
- [ ] 単体目標線（赤/オレンジ/緑破線）と均等努力目標線（青点線）が**両方**表示
- [ ] ラベル位置が重ならず読み取れる

### C. フェーズ構成タブ
- [ ] 6F選択時、C群が0%ではなく適切な値（20人前後）
- [ ] What-if スライダー（月間入院数・目標稼働率）が病棟別では60-90人レンジ
- [ ] Little法則の計算プロセスが expander で表示

### D. 朝のブリーフィング / 意思決定ダッシュボード
- [ ] 5F・6F の rolling LOS が**並列**表示
- [ ] 6Fが基準超過（22.6日 > 21日）時にアラート発動
- [ ] 施設基準判定は**各病棟ごと**で、病院全体ではない

### E. クリア計画
- [ ] 「全体」モード時、超過病棟（6F）のクリア計画のみ表示
- [ ] 「C群追加退院3名」など現実的な指示内容

### F. HOPE送信サマリー
- [ ] 5F/6F個別の rolling LOS 入力欄が表示
- [ ] 400文字制限内で生成される

### G. 均等努力メッセージ
- [ ] Δ > 0 の時：「5F→91%、6F→95.3%、均等努力で目標達成」メッセージ
- [ ] Δ ≤ 0 の時（仮想）：「既に目標達成ペース、追加努力不要」メッセージ
- [ ] 稼働率は**経営目標**、施設基準は**平均在院日数のみ**と明記

## 重要な不変条件（Invariants）

以下が崩れたら**必ず**リグレッション：

1. **均等努力 Δ > 0**: デモデータの核。負になるとストーリーが破綻する。
2. **5F単体目標が「困難」、6F単体目標が「達成可能」**: 単一病棟では足りず協力が必要というメッセージ。
3. **6F rolling LOS > 5F rolling LOS**: 6Fが drifting しているストーリー。
4. **施設基準 = 平均在院日数のみ**: 稼働率は経営目標であって施設基準ではない。UIメッセージで混同しない。
5. **過去90日データはrolling LOSのみに使用**: 日次推移チャートや月次集計に混入しない。

## 意図的に変更するとき

スモークテストの EXPECTED を更新する際は、必ず：
1. 変更理由をコミットメッセージに記載
2. 対応する UI/ドキュメント/シナリオ台本を連動更新
3. 新しい期待値が上記「不変条件」に反していないか確認
