2なぜ GitHub を使うのか

「Mac のフォルダだけで十分では？」と思うかもしれません。GitHub を使うメリットを比較してみましょう。

項目 Mac だけ GitHub も使う
バックアップ Mac が壊れたら終わり 危険 クラウドに自動保存 安全
変更履歴 上書き保存で元に戻せない 全ての変更を記録、いつでも巻き戻し可能 安心
共同作業 ファイルをメールで送る 同じ場所を見て一緒に作業できる 効率的
AIレビュー 自分で確認するだけ 変更をAIが自動チェック 品質向上
事故防止 間違えたファイルを共有しがち 承認プロセスを挟める 安全
☁️ イメージ iCloud のようなバックアップ機能に加えて、変更履歴の完全記録と共同作業の仕組みが付いたもの、と考えてください。

3基本用語を病院に例える

Git / GitHub の用語は聞き慣れないものばかりですが、病院業務に置き換えると理解しやすくなります。

リポジトリ
repository
プロジェクト全体の保管場所
🏥 病院例：診療科のカルテ棚。全ての記録が一箇所にまとまっている。
コミット
commit
変更を1回分記録すること
📝 病院例：1回分のカルテ記載。日時と誰が書いたかも自動で記録される。
コミットメッセージ
「何を・なぜ変更したか」の一行メモ
📋 病院例：SOAP の見出し。後から見返すとき一目でわかるように。
ブランチ
branch
本体に影響を与えずに作業する「別紙」
📄 病院例：正式マニュアルの改訂案を、別紙で書くイメージ。正式版はそのまま安全。
マージ
merge
別紙で書いた改訂案を正式版に反映すること
✅ 病院例：改訂案の承認が降りて、正式マニュアルに差し替える。
push
git push
手元の変更をクラウド（GitHub）に送る
⬆️ 病院例：改訂した手順書を共有フォルダにアップする。
pull
git pull
クラウド（GitHub）から最新の状態を手元にもらう
⬇️ 病院例：共有フォルダから最新の手順書をダウンロードする。

4PR（Pull Request）とは

PR は「この変更、本体に入れていいですか？」という確認ステップです。

病院で例えるなら、マニュアル改訂案を提出して上司に「この内容で問題ないですか？」と確認を取るプロセスです。

一人でも PR を使う理由

  • AI が自動チェック — Claude が変更内容をレビューして問題を発見してくれる
  • 変更の記録が残る — 「いつ・何を・なぜ変えたか」が一覧で見られる
  • 事故防止 — うっかり本体を壊してしまうリスクを減らせる
  • 振り返りが容易 — 後日「あの変更なんだっけ？」と調べやすい
🔍 豆知識 PR は「Pull Request（プルリクエスト）」の略。「変更を取り込んでください (pull してください)」というリクエストなので、この名前が付いています。

5GitHub Actions — 自動化の仕組み

GitHub Actions は「〇〇が起きたら自動で△△する」という仕組みです。

病院での例え

自動オーダリングシステムを想像してください。

  • 「入院オーダーが出たら → 自動で採血セットを組む」
  • 「体温38.5度以上 → 自動で血液培養オーダーを提案」

GitHub Actions も同じ考え方です：

  • 「PR が出されたら → 自動で Claude がレビューする」
  • 「main に変更が入ったら → 自動でテストを実行する」
⚙️ 今日設定したもの PR が来たら Claude が自動でレビューする仕組み（claude-review.yml）を作りました。

6今日やったこと（実践の振り返り）

今日の作業をステップバイステップで振り返ります。

claude-review.yml を作成
GitHub Actions の設定ファイルを作成。「PR が出されたら Claude が自動レビューする」というルールを定義しました。
ANTHROPIC_API_KEY を GitHub Secrets に登録
Claude が動くために必要な「鍵」を GitHub の金庫（Secrets）に安全に保管しました。パスワードのようなもので、誰にも見られない場所に保存されます。
テスト PR で動作確認 → 認証エラー発生
最初の実行で認証エラーが発生。GitHub Actions が GitHub にアクセスするための権限（github_token）が不足していたことが原因でした。
github_token を追加して修正 → 成功
不足していた権限を追加設定したところ、Claude の自動レビューが正常に動作しました。
REVIEW.md（レビューガイドライン）を作成・マージ
Claude がレビュー時に参照するチェック項目をまとめたファイルを作成し、正式版（main ブランチ）に反映しました。
自動レビューの仕組みが完成
以後、PR を出すだけで Claude が自動レビューしてくれる環境が整いました。
💪 振り返りのポイント 最初はエラーが出ましたが、原因を特定して修正するプロセスも含めて、これが実践的な学習です。エラーは「失敗」ではなく「次の設定が必要」というサインです。

8gh コマンドの仕組み

Claude Code が GitHub と連携するとき、実は特別な接続を持っているわけではありません。

仕組みはとてもシンプルです：

  • 先生の Mac には gh（GitHub CLI）というアプリがインストールされている
  • 場所は /opt/homebrew/bin/gh
  • Claude Code は、この gh コマンドを「借りて」GitHub にアクセスしている
🏠 例えるなら 秘書（Claude Code）が「先生、共有フォルダのパスワード使っていいですか？」と許可をもらって作業しているイメージです。秘書自身が鍵を持っているのではなく、先生の Mac にある鍵を借りているだけです。

そのため、gh が Mac にインストールされていて、GitHub にログイン済みであることが前提条件になります。
