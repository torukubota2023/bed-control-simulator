## 学習した概念

### 1. PR（Pull Request）とは

「この変更、本体に入れていい？」という確認ステップのこと。

**クラスの共有ノートの例え:**

1. 共有ノート（main）がある
2. 自分用のコピー（ブランチ）を作って書き足す
3. 「これ追加していい？」とお願いを出す ← **これがPR**
4. チェックする人が「いいね」or「ここ直して」と確認
5. OKなら共有ノートに反映（マージ）

**ポイント**: PRは「変更の提案 + レビューのプロセス」をセットにした仕組み。mainブランチを直接変更するのではなく、必ずこのステップを挟むことで品質を保つ。

---

### 2. PRのレビュアーは誰か（3パターン）

| パターン | レビュアー | 適したケース |
|----------|-----------|-------------|
| パターン1 | 人がチェック | チーム開発なら同僚やリーダー |
| パターン2 | AIがチェック | GitHub ActionsにClaude Code Actionを設定 |
| パターン3 | 両方（AI→人の二段階） | 大規模チーム・高品質要求 |

**一人運用なら**: パターン2（AIレビュー）が最も実用的。自分では見落としがちなバグやコード品質の問題をAIが指摘してくれる。

---

### 3. GitHub Actions とは

「〇〇が起きたら自動で△△する」仕組み。

**例え**: ドアが開いたら → 自動で電気がつく、と同じセンサーのような自動化。

- **トリガー**: PRが作られた、pushされた、スケジュール実行、など
- **アクション**: テスト実行、コードレビュー、デプロイ、通知、など
- **設定方法**: `.github/workflows/` ディレクトリにYAMLファイルを置くだけ

---

### 4. Claude Code Action の設定

`.github/workflows/claude-review.yml` を作成することで、PRが作られると自動でClaudeがレビューする仕組みを構築できる。

**必要なもの:**
- `ANTHROPIC_API_KEY`（GitHub Secretsに登録）
- `GITHUB_TOKEN`（GitHub Actionsが自動で提供するが、permissionsの設定が必要）

**動作の流れ:**
1. PRが作成される（トリガー）
2. GitHub Actionsが起動
3. Claude Code Actionがコードを読み取り
4. レビューコメントをPRに投稿

---

### 5. GitHub Secrets

パスワードやAPIキーを安全に保管する場所。コードに直接書くと漏洩リスクがあるため、Secretsに登録して参照する。

**設定方法:**
1. リポジトリの Settings を開く
2. Secrets and variables → Actions を選択
3. New repository secret をクリック
4. Name: `ANTHROPIC_API_KEY` / Secret: `sk-ant-...` で始まるキーを入力

**重要**: Secretsに登録した値は、登録後は誰にも見えなくなる（上書きのみ可能）。

### 6. Claude Code と GitHub の関係（アーキテクチャ）

```
┌─────────────────────────────────────┐
│      先生のMac（ローカル）             │
│                                     │
│  Claude Code ← 先生と対話しながら作業  │
│      ↓                              │
│  ファイルを読む・書く・編集する         │
│      ↓                              │
│  git コマンドで GitHub とやり取り      │
└──────┬──────────────────────────────┘
       │ push / pull
       ↓
┌─────────────────────────────────────┐
│    GitHub（リモート・クラウド）         │
│                                     │
│  ファイルの保管・履歴管理・PR・Actions  │
└─────────────────────────────────────┘
```

**理解すべき核心:**
- **Claude Code** は「ローカルで動く作業員」— 先生のMac上でファイルを直接操作する
- **GitHub** は「保管する倉庫」— ファイルの履歴管理、共有、自動化の基盤
- Claude Codeが直接GitHubを編集しているわけではなく、**ローカルで編集 → push で送信**という流れ

### 7. git の基本操作

| コマンド | 意味 | 例え |
|----------|------|------|
| `git add` | 変更を記録対象にする | 「これ提出します」と手を挙げる |
| `git commit` | 変更を確定する | ノートに日付とメモを書いて綴じる |
| `git push` | GitHub に送る | 図書館に提出する |
| `git pull` | GitHub から取得 | 図書館から最新版を借りてくる |
| `git branch` | 枝分かれを作る | ノートのコピーを作って別の内容を試す |
| `git status` | 現在の状態を確認 | 「今どうなってるっけ？」と確認 |
| `git log` | 履歴を見る | ノートの更新履歴を見返す |

**基本の流れ:**
```
作業する → git add → git commit → git push → PRを作る → レビュー → マージ
```

### 8. GitHub の主な機能

| 機能 | 説明 | 用途 |
|------|------|------|
| **リポジトリ** | プロジェクト単位の保管庫 | コード・ドキュメントの保管 |

| **Issues** | やること・バグ報告の管理 | タスク管理、バグトラッキング |
| **Secrets** | APIキーなどの安全な保管 | 認証情報の管理 |

### 9. gh コマンドの仕組み

- Claude Code は `gh`（GitHub CLI）を使ってGitHubのAPIを叩いている
- 特別な設定をしたわけではなく、MacにHomebrewでインストール済みの `gh` を借りているだけ
- 場所: `/opt/homebrew/bin/gh`
- `gh` コマンドで PR の作成、Issue の管理、Actions の確認などが可能

**よく使われるコマンド例:**
```bash
gh pr create --title "タイトル" --body "説明"   # PR作成
gh pr list                                      # PR一覧
gh pr view 2                                    # PR #2の詳細
gh run list                                     # Actions実行一覧
gh run view <run-id>                            # 実行結果の詳細
```

## トラブルシューティングの知見

### GitHub Actions が失敗する場合
1. **認証エラー**: `GITHUB_TOKEN` の permissions が正しく設定されているか確認
2. **APIキーエラー**: GitHub Secrets に `ANTHROPIC_API_KEY` が正しく登録されているか確認
3. **タイムアウト**: Actions の実行時間制限を確認（デフォルトは6時間だが、早期に失敗する場合は設定問題）

### よくある誤解
- Claude Codeが直接GitHubを操作しているわけではない（ローカルのgitコマンド経由）
- `gh` コマンドは特別なものではなく、Homebrewでインストールされた通常のCLIツール
- CLAUDE.mdはGitHubにpushすれば、ローカルのClaude CodeだけでなくClaude Code Actionも読む
